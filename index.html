NewPMJobs Product Journey

A personal localhost project became a production multi-user site, built entirely with Claude Code and a dozen other AI tools. Every change here was a deliberate product decision, and every decision taught me something.

The 30-second version

The 30-second version

NewPMJobs is a daily job-alert tool for product managers: pick the companies you want to work for, and every morning it emails you the new, relevant PM roles, pulled from each company's own careers page and filtered down to what actually fits. The part worth knowing is who built it and how.

  • One product, end to end: scraping, a self-healing data pipeline, a web app, a daily email and full monitoring, all live and serving real users.
  • Built by a PM, coded by AI: I made every product, design and architecture decision, and AI tools wrote the code under that direction.
  • Reliability treated as a feature: most of the effort went into catching silent failures (a broken source, a quiet zero, a missed email) before a user ever notices.
  • The proof in numbers: page speed went from 49 to 100, sub-second data calls replaced multi-minute browser scrapes, and the catalog grew to 519 companies across 7 hiring platforms.

1. Concept

Where the whole thing started: one annoying daily chore and the smallest script that could kill it.

Phase 1: The Localhost MVP

Goal: Stop manually checking career pages every morning.

I was spending 30+ minutes a day visiting individual company career sites, searching for PM roles and trying to remember which listings were new. The idea was simple: write a script that does the checking for me and emails me when something new shows up.

What I Built

An Express backend with a Puppeteer scraper that hit a handful of Greenhouse career pages, filtered jobs by PM-related keywords, stored results in a PostgreSQL database and ran on my local machine. No frontend. No authentication. Just a script I could trigger manually and an email that would land in my inbox.

Stack: Express, Puppeteer, PostgreSQL (Supabase), Resend for email.

What I Learned

Start with the smallest version that solves your problem. The first version had no UI, no multi-user support and only knew about one type of career page. But it answered the only question that mattered: "Are there new PM roles today?" That was enough to validate that the tool was worth building further.

Greenhouse has a public data feed most people do not know about. Instead of loading the page in a browser, I could hit a structured public board endpoint and get clean JSON back. Faster, more reliable and no browser required. This insight shaped the entire scraper architecture going forward: always prefer structured feeds over reading rendered HTML.

Drilling deeper on the "why"

The real reason to start at localhost was not laziness, it was that I did not yet know whether the problem was worth solving with software at all. Thirty minutes a day felt annoying, but feelings are a bad spec. A script I could run by hand for a week told me something a polished product never could: whether I would actually open the email every morning. Validating the habit was the point; the code was just the cheapest way to test it.

Alternative considered I could have started with a browser extension or a no-code scraper service and skipped writing a backend at all. I chose a real Express + database stack from day one because the thing I most wanted to learn was whether new-versus-seen tracking held up over days, and that needs persistent state. A no-code tool would have validated "can I see jobs" but not "can I trust this to remember what it already showed me," which was the actual product.
What this unlocked Discovering Greenhouse's structured feed on day one is the decision the next thirty phases inherit. Every "API-first, browser as a last resort" call later in this journey traces back to that first morning. Picking the reliable data path before I had any users meant the architecture never had to be unwound under load.

2. Setting the Foundation

Getting the tool off my laptop, onto the open web and able to cover the companies real users care about.

Phase 2: Going Live

Goal: Access the tool from anywhere, not just my laptop.

Running on localhost meant my computer had to be on. If I was traveling or on my phone, I could not check results. The tool needed to be a real website.

What I Built

Deployed the backend to Railway (auto-deploys from git push) and built a Next.js frontend on Vercel. Set up a custom domain on Cloudflare. Added a Railway Cron job to trigger the daily scrape automatically every morning, so I did not need to run anything manually.

Stack additions: Next.js 16 (Vercel), Railway Cron, Cloudflare DNS.

Key Decisions

Railway + Vercel instead of a single platform. The backend needs Puppeteer (a headless browser), which is resource-heavy and does not run well on serverless. Railway gives me a persistent container. The frontend is static-ish Next.js, which Vercel handles perfectly. Splitting them added cross-origin complexity but let each layer use the best tool.

Cloudflare DNS-only mode (grey cloud). I initially tried Cloudflare's proxy for SSL, but both Vercel and Railway manage their own SSL certificates. The proxy caused certificate conflicts. Lesson: when your hosting provider handles SSL, keep the CDN in DNS-only mode.

What I Learned

Deploy timing matters more than you would think. After pushing code, Railway takes about 60 seconds and Vercel about 30 seconds to deploy. I kept testing too early and seeing old behavior. Built in a habit of waiting 90+ seconds after every push before verifying.

Cross-origin requests with www redirects are a trap. Vercel redirects the root domain to the www subdomain. My backend only allowed the root origin, so authenticated requests from www. were getting blocked. The fix was to auto-detect and allow both origins from a single frontend-URL config value.

Phase 3: Covering The Companies I Actually Care About

expanding scraper coverage

Goal: Cover the companies I actually care about, not just the ones on Greenhouse.

The MVP only worked for Greenhouse companies. But companies I wanted to track (Stripe, Google, OpenAI, Slack, Netflix) used different ATS platforms or entirely custom career pages.

What I Built

A multi-platform scraper that supports seven different approaches:

  • Greenhouse API for DoorDash, Discord, Reddit, Instacart, Figma, Airbnb, a16z, Twitch
  • Lever API for any company on Lever
  • Ashby GraphQL for OpenAI and others on Ashby
  • Workday JSON API for Slack, Salesforce and others on Workday
  • Eightfold API for PayPal
  • Custom API handlers for Uber, Netflix, Atlassian (each has a unique API format)
  • Browser-based fallback for Stripe, Google and unknown sites

Built a 7-layer platform auto-detection engine so that when someone pastes a careers URL, the system automatically figures out which ATS is behind it: known hostnames, direct ATS URLs, embed detection, rendered-page detection, structured-feed probes and a generic fallback. Added post-scrape validation that quality-scores every run: catches zero results, vague locations, duplicates and invalid URLs.

Key Decisions

API-first scraping with the browser as a last resort. Structured-feed scrapes finish in under 1 second. Browser-based scrapes take 2 to 3 minutes because they load real pages. For the daily cron, this difference matters enormously. Every new company I add, I first check if there is a structured feed available.

17 PM-specific keywords instead of broad scraping. I could return all jobs, but that would bury the signal. Filtering by keywords like "product manager," "product lead" and "product ops" keeps results relevant. The tradeoff: edge cases slip through. When a16z showed zero PM roles despite having many jobs, I discovered they use "Product Growth" as a title, which was not in my list. Added it.

What I Learned

Every ATS has quirks. Salesforce's careers domain redirects to a marketing page with zero jobs; the real board is a Workday URL at a completely different domain. Reddit's Greenhouse board needs department-level fetching plus keyword filtering. Each new company taught me something about how career pages actually work in practice.

Stale data is a silent killer. When I added a proper API scraper for PayPal, the old browser-scraped jobs were still in the database with garbage data (title, location and date mashed into one field). I learned the hard way: after changing a scraper, delete the company and re-add it to flush the stale data.

Drilling deeper on the "why"

The auto-detection engine is not really about convenience. It is about who pays the cost of a wrong guess. If I hand-classified every company's platform, then every misclassification became a silent zero that I would only notice days later. By making detection a layered probe that tries the cheap, structured paths first and degrades to the browser only when everything else fails, the system fails toward "slow but correct" instead of "fast but wrong." The seven layers are really seven progressively-less-confident guesses, and the architecture trusts the most confident one that actually returns jobs.

Alternative considered I could have written one general browser scraper that reads the rendered page for any company and skipped the per-platform integrations entirely. One code path, far less to maintain. I chose the multi-platform approach because the browser path is 100 to 180 times slower and breaks whenever a page changes its layout. A single generic scraper would have made the daily cron unfinishable as the catalog grew, and it would have turned every front-end redesign at every company into an outage. The integrations cost more upfront and paid it back every single day the cron ran.
Edge cases that bit The gnarly boundary cases were the companies that lie about what they are. Some host their careers page on a custom domain but actually run a standard ATS underneath, so the obvious detection misses them and they fall through to the slow path. The fix was a speculative probe: extract the company slug from the hostname and try it against the known structured feeds in parallel before giving up to the browser. The other trap was a company with many jobs and zero PM roles, which looks identical to a totally broken scraper. That ambiguity is small here, but it grows into its own multi-phase problem later (Phases 15 and 21).
What this unlocked Seven scraper types and a detection engine turned "add a company" from a code change into a paste-a-URL action. That is the precondition for everything downstream: the check-then-add UX, the multi-user catalog and eventually letting users add companies themselves. Coverage stopped being a bottleneck on growth.

Phase 4: Turning It Into A Real Product Other People Can Use

authentication and multi-user architecture

Goal: Let other people use the tool, not just me.

The original version had no login. Every company was mine, every job was mine. To share the tool, I needed authentication, data scoping and a way for each user to track their own set of companies.

What I Built

Authentication via magic links. No passwords to store, no reset flow, no brute force to worry about. User enters email, gets a link, clicks it, done. Built on Supabase Auth with Resend as the SMTP provider.

Shared company catalog with per-user subscriptions. Instead of each user having their own copy of "Uber" (scraped separately), there is one shared entry. Users subscribe to the companies they want to track. One daily scrape serves all users, and scraping cost scales with the number of companies, not users.

New database tables for user subscriptions, per-user bookmarks, user preferences (email frequency: daily, weekly or off) and a rate-limited record of new-company submissions (10 per user).

Admin dashboard with stats, scrape issue reports, help submissions and user management, restricted to a single admin email. Rate limiting on all API routes: tighter on writes, looser on reads.

Key Decisions

Shared catalog vs. per-user companies. This was the biggest architectural decision in the project. Per-user would be simpler, but it would mean scraping the same company once per user. With a shared catalog, scraping cost is proportional to companies, not companies times users. The tradeoff is that users cannot customize which roles they see per company, but PM roles are PM roles regardless of who is looking.

Magic links instead of passwords. The security surface is dramatically smaller: no credential storage, no resets, no brute force. The tradeoff is that login requires email access, which adds friction on mobile. For a tool where users log in once and then get daily emails, the reduced surface was worth it.

Job archival instead of hard deletion. When a job disappears from a careers page, I mark it "removed," and after 60 days it moves to "archived." This preserves favorited jobs. Slightly more storage, but losing someone's bookmarks is a bad experience.

What I Learned

Server-stored login cookies create a frustrating browser limitation. Supabase Auth stores the login token in a cookie JavaScript cannot read, which is good for security. But my frontend needs the token to call the backend. The fix: a Next.js server route that reads the cookie server-side and returns the token; the client caches it in memory.

The PKCE magic link flow breaks across devices. My first auth used PKCE, which stores a verifier in the browser where you requested the link. Open the link in a different browser or your email app's built-in viewer, and the verifier does not exist, so login silently fails. Fixed by switching to a token-hash flow that works from any device.

Service key vs. anon key is a subtle, critical distinction. Supabase has two keys: one that respects per-row access rules and one that bypasses them. My backend used the wrong one, and the current-user function returned NULL for every query, making all user-scoped queries return empty. The error was invisible because the queries succeeded with zero results instead of throwing.

N+1 queries sneak in during multi-user rewrites. Subscribing a user to N companies first ran 2N sequential database queries. Batching them into parallel operations cut response time dramatically. I now audit every new feature for this pattern.

Drilling deeper on the "why"

The shared catalog was not chosen for elegance. It was chosen because the unit economics of the wrong model break the product silently. With per-user companies, the daily scraping work multiplies by every new signup. The product would have felt fine at ten users and quietly become unaffordable, and slow, at a few hundred, exactly when growth is the goal. Decoupling cost from headcount up front is what let the catalog grow into the hundreds later without the daily cron getting more expensive each time someone joins.

Alternative considered The simpler build was a copy of each company per user, scraped independently. It needs no subscription table and no shared-state reasoning, and it lets each user filter roles however they like. I rejected it because it makes the most important cost in the system, daily scraping, scale with users times companies instead of just companies. The thing that makes per-user "simpler" to write is the same thing that makes it impossible to afford. I traded a more complex data model for a cost curve that does not punish success.
Edge cases that bit Three boundary failures defined this phase, and all three were silent. The wrong database key made every user-scoped query return an empty list with no error, so the app looked like it had no data rather than a misconfiguration. The PKCE flow failed only when the link was opened somewhere other than where it was requested, which is exactly what happens when an email app opens links in its own viewer, so it passed every test on my own machine and broke for real users. And the N+1 pattern only showed up under a real subscribe action, not a single-company test. The common thread: each one was invisible on the happy path and only appeared at a boundary I had not exercised.
What this unlocked This phase is the hinge of the whole project: it is the moment the tool stopped being mine and became a product other people could use. The shared-catalog decision specifically is what made the daily scrape a fixed cost the rest of the journey could build features on top of, instead of a tax that grew with every user.

The product taking shape: a safe add-flow, salary data, a real landing page, speed and a pre-launch security pass.

Phase 5: A Safe Way To Add A Company

the check-then-add flow

Goal: Do not let users add a company and get garbage results.

The original "add company" flow was: paste URL, click Add and hope the scraper finds PM roles. If it did not work, the company was already in the database with zero jobs and the user had no idea why.

What I Built

A preview-before-commit flow with four states:

  1. Input: User pastes a careers URL (company name auto-detected from the URL)
  2. Checking: Animated progress (the scraper runs but saves nothing to the database)
  3. Preview: Shows the detected name, sample job titles and locations and "Found X PM roles. Does this look right?"
  4. Confirm or retry: "Yes, Add It" saves to the database. "No, Try Again" lets the user fix the URL and provide feedback. "Cancel" after feedback files it as a help submission for admin review.

Also built a company name auto-detection system that recognizes 40+ known hostnames, falls back to ATS slug extraction, then to generic hostname parsing. If the URL matches an existing company, the system surfaces it and offers to subscribe instead of creating a duplicate.

Key Decisions

Preview without saving. The check endpoint runs the full scrape pipeline but returns results without touching the database. Only when the user confirms does the save fire, passing the pre-checked jobs along so the scraper does not run twice.

URL-based dedup instead of name-based. Early versions matched duplicates by name, but "Open ai" vs. "OpenAI" vs. "openai" caused false negatives. Switching to URL domain extraction is much more reliable.

What I Learned

Error recovery UX matters more than happy-path UX. Most users hit the happy path and never think about it. But when the scraper returns zero or wrong results, the user needs a clear way to understand and retry. The "retry with feedback" state turned a dead end into a recoverable flow, and the feedback gave me real data on which companies needed scraper improvements.

Auto-detecting company names is surprisingly hard. The same ATS uses different URL patterns. Each needs different parsing to extract a clean name. The 40+ hostname lookup table was built one company at a time as I hit new edge cases.

Phase 8: Making It Load Fast

performance optimization

Goal: Ship a landing page that loads fast, especially on first visit.

After building the landing page, Lighthouse desktop score was 49. Too much JavaScript loaded upfront: 1,500+ lines of landing code, brand-color calculations on every render and analytics blocking the main thread before first paint.

What I Built

Code-splitting. Split the 1,528-line landing page into above-fold hero (loads immediately) and below-fold sections (lazy-loaded). Users see the hero instantly while the rest loads in the background.

Pre-computed brand colors. The page blended brand colors toward white for card backgrounds, about 80 runtime calculations per render. Moved the computation to module-load time so it runs once.

Deferred analytics. The analytics init ran at top-level module scope, blocking the main thread before first paint. Moved it into an effect so it runs after the page is visible.

Result: Desktop Lighthouse went from 49 to 100 (Total Blocking Time dropped to 0ms). Mobile stayed at 72 to 77, bottlenecked by the React runtime's 225KB, which is beyond what I can optimize without a bigger architectural change.

What I Learned

Measure before optimizing. The three fixes were guided by Lighthouse diagnostics. Without measuring, I might have guessed wrong. The analytics blocking, for example, was not obvious until I looked at the Total Blocking Time breakdown.

There is a ceiling set by your framework. On mobile the bottleneck is the React runtime itself. No amount of code-splitting fixes that; the next step would be a larger architectural change.

Pre-computation is underrated. Moving brand-color math from render-time to load-time eliminated ~80 calls per render with a one-line change. Always check: is this computation the same every time? If yes, compute it once.

Phase 9: Getting Safe For Real Users

security hardening

Goal: Get the app ready for real users who are not me.

Before sharing the URL publicly, I ran a security audit. Single-user tools can get away with shortcuts. Multi-user tools cannot.

Security Headers

Added five security headers to all frontend responses: Content-Security-Policy (a whitelist of allowed sources, built dynamically from config), Strict-Transport-Security (forces HTTPS), X-Frame-Options: DENY (blocks clickjacking), X-Content-Type-Options: nosniff and Referrer-Policy.

Full Security Audit

A set of fixes deployed together:

  1. Open redirect prevention. A post-login redirect parameter could be pointed at an external site. Fixed by validating it is a relative path only.
  2. XSS in help emails. User-submitted feedback was embedded in admin emails without escaping, so a user could inject code that ran when I opened the email. Added escaping to all user input before it goes into templates.
  3. Hardcoded admin email. Extracted to a shared constant read from config.
  4. PII in server logs. User email addresses were logged during the daily cron. Replaced with truncated user IDs.
  5. Scraper timeout. The generic browser scraper had no timeout, so a hanging page could block the entire daily cron. Added a 120-second timeout.
  6. Cron overlap guard. Added a flag to prevent two scrapes running simultaneously.

Input Validation and SSRF Protection

  • UUID validation on all ID parameters
  • HTTPS-only URLs when adding companies
  • LinkedIn URLs blocked (their career pages never scrape correctly)
  • Private IP ranges blocked to prevent SSRF, where someone could use the scraper to probe internal networks

DNS Security

Configured DMARC, DKIM and SPF records so alert emails are less likely to land in spam and harder to spoof.

What I Learned

Security is not a feature you add at the end; it is a list of things that are always wrong. Every one of these existed from day one. None caused problems while I was the only user. All would have been exploitable the moment I shared the URL.

CSP is powerful but painful with this framework. The framework injects inline scripts and styles for hydration, which means a strict CSP needs nonce-based exceptions. For now an inline-script allowance is required; nonce-based CSP is a future upgrade.

Drilling deeper on the "why"

The reason to audit before launch rather than after is that the cost of every one of these bugs is zero until the exact moment it is catastrophic. A single-user tool with an open redirect is harmless; the same code, the day a stranger visits, is a phishing vector pointed at my own domain. Security debt does not accrue interest gradually, it sits at zero and then steps straight to its full value the instant the audience changes. Doing the audit as the gate to going public is the only point where the work is cheap and the risk is still hypothetical.

Alternative considered I could have launched first and hardened reactively, fixing issues as they surfaced or as users reported them. That is the faster path to "live." I chose to gate the public URL behind the audit because the failure mode of reactive security is that you find out about the hole from the person exploiting it, not from a report. For a solo builder with no security team watching logs in real time, a clean pre-launch sweep is the only review the app was realistically going to get before strangers arrived.
Edge cases that bit The scariest boundary case was SSRF. My product is, by design, a thing that fetches arbitrary URLs a user hands it. That is the exact shape an attacker wants: submit an internal address instead of a careers page, and the server fetches it on their behalf, potentially reaching systems that are not supposed to be reachable from the outside at all. The defense is to treat every user-supplied URL as hostile and refuse private address ranges outright. The two other quiet ones: a browser scraper with no timeout can hang the entire daily run on one slow page, and the cron with no overlap guard can run two copies of itself and fight over the database.
What this unlocked This phase is what made "share the link" a safe action rather than a gamble. It also set the template every later audit follows: list the things that are silently wrong, fix them in one pass, and write down why. The SSRF and input-validation rules in particular became permanent invariants the rest of the product is built inside.

Phase 12: Making The Daily Run Dependable

cron reliability and operations

Goal: Make sure the daily scrape actually completes, every day, without manual intervention.

What I Built

Await-based cron endpoint. The cron trigger must wait for the daily check to finish before responding. My original version returned immediately, and Railway's auto-sleep would kill the process before scraping finished, causing missed alerts on quiet weekends.

Single-source scheduling. Removed an in-process scheduler that caused duplicate runs alongside Railway's cron. Only Railway Cron initiates the daily check.

Overlap guard. A flag prevents a new run starting if the previous one has not finished.

120-second timeout on generic scraper pages that hang, so one bad page does not block the whole run.

What I Learned

Auto-sleep infrastructure changes the rules. Railway keeps the backend running while it receives requests but sleeps after inactivity. Fire-and-forget patterns that work on always-on servers fail silently here. The cron endpoint has to hold the connection open until the work is done.

Duplicate runs are worse than missed runs. A missed scrape means yesterday's data. A duplicate means two processes fighting over the database, potentially creating duplicates or double emails. The overlap guard was added after I saw double emails during testing.

Parallel query execution. The company detail endpoint runs 3 queries in parallel; the dashboard runs 2.

Database-level filtering. The "new jobs today" calculation runs in PostgreSQL rather than in JavaScript.

Phase 15: Scrapers That Fix Themselves

self-healing scrapers

Goal: Stop scraper failures from being invisible until a user complains.

What I Built

Companies change their job board platforms without warning. When that happens, the daily scrape silently returns zero results, and nobody knows until someone checks their dashboard days later.

I built an auto-remediation system into the daily cron. When a scraper returns zero jobs for a non-custom company, the system automatically runs platform detection to check if the company switched providers. If it finds a new platform, it updates the database, re-scrapes with the correct scraper and tracks the fix. The daily admin email now has two sections: a green "auto-fixed" section and a red "still needs attention" section.

I also fixed a subtle false-alarm problem. Most scrapers pre-filter jobs by PM keywords internally, which means zero results could mean "no PM roles right now" or "the scraper is completely broken." Now, actual failures throw exceptions (caught by error handling), while zero PM results are treated as valid outcomes. This eliminated phantom alerts for companies that simply had no PM openings.

Key Decisions

Auto-remediate only for known ATS platforms, not custom scrapers. Companies with bespoke scraping logic cannot be auto-detected, so a human looks at those. But for the many companies on standard platforms, switches are detectable and fixable automatically.

Two-tier email instead of a failure dump. The old email was a wall of red. The new one separates auto-fixes (green, informational) from real problems (red, actionable), so I can glance at it and know whether I need to do anything today.

Removed the zero-results alert entirely. My first instinct was to alert on zero PM jobs. But after investigating four "failures," two were correct: those companies just were not hiring PMs. Alerting on ambiguous signals creates noise that trains the operator to ignore alerts.

What I Learned

The hardest bugs look like correct behavior. A zero result looks exactly like "this company is not hiring PMs." The only way to tell that apart from "the scraper is pointed at the wrong platform" is to re-run platform detection. That is why the auto-remediation step exists.

Pre-filtering creates ambiguity that post-filtering does not. If every scraper returned ALL jobs and filtered afterward, I could distinguish "200 jobs, none PM" from "scraper returned nothing." Pre-filtering is faster but makes the output ambiguous. I kept it for performance and added the remediation layer to handle the ambiguity.

Alert fatigue is worse than missing alerts. Four alerts a day for non-issues would train me to stop reading the email within a week. Reducing to genuine, actionable failures keeps the signal-to-noise high.

Drilling deeper on the "why"

The deeper realization here is that "zero results" is not one event, it is two completely different events wearing the same costume, and my monitoring could not see the costume. A scrape that succeeds and finds no PM roles and a scrape that is broken and finds nothing produce byte-identical output. No amount of better logging on the zero itself can separate them, because the information that distinguishes them does not live in the result, it lives in whether the platform still exists. So the fix was not a smarter alert, it was an active probe: re-run detection and let the answer to "did the platform move?" disambiguate the zero. The lesson generalizes to anything where absence of a signal is ambiguous, you have to go get the disambiguating fact, not stare harder at the absence.

Alternative considered The obvious move was to alert on every zero-PM company and let me investigate each one by hand. It guarantees I see every real break. I rejected it after the first day of using it, because two of the first four alerts were companies that were genuinely just not hiring PMs. An alert channel that cries wolf two times out of four is one I will have learned to ignore within a week, which means it fails me precisely on the fifth alert that is real. I chose to alert only on unambiguous failures (exceptions) and to make the system itself resolve the ambiguous ones, trading some coverage for an alert channel I will actually still be reading in a month.
Edge cases that bit The cleanest example was a company that migrated its entire ATS from one platform to another. The scraper kept hitting the old, now-dead endpoint and returning zero, which looked identical to "they stopped hiring PMs." Only re-running detection revealed they had moved, at which point the system could repoint itself and recover the listings on its own. The boundary that made this hard is that the broken state and the healthy-empty state are indistinguishable from the outside; the only tell is an action, not an observation.
What this unlocked This is the first time the system started healing itself instead of paging me, and it set the direction for everything operational that follows. The two-tier "fixed vs. needs-attention" email became the template, and the core insight, that a silent success is more dangerous than a loud failure, becomes the spine of Phases 21, 29 and 31.

Phase 16: Stopping Bad Data At The Source

the data quality pipeline

Goal: Stop bad data from reaching users, and give me a daily health report so I catch issues before anyone else does.

What I Built

Microsoft was showing 157 PM jobs in the daily email, dozens from India, the UK and elsewhere. The root cause was embarrassing: location filtering only existed as a frontend toggle. The backend was storing every global PM job and including all of them in email alerts. The "US Only" button hid the problem in the UI, but the email had no such filter.

I ported location detection to the backend and added a second filter pass in the validation pipeline. Every job now goes through two gates: does the title match PM keywords, and is the location in the US? I added explicit non-US patterns (60+ countries) instead of just checking for US matches. If a location matches nothing either way, it gets excluded by default. Safer to miss an unusual US location string than to email someone jobs from another continent.

The second part was a daily quality evaluation. After every cron scrape, the system runs a quality check across every company and emails me a scorecard: US jobs, non-US filtered out, change from yesterday, quality score and any flags. Problem companies sort to the top.

Key Decisions

Filter at the validation layer, not in each scraper. Every scraper flows through one validation step before anything is saved. Adding the location filter there applies it universally with zero changes to individual scrapers. The alternative (filtering in each of the 8+ scrapers) would have been fragile and easy to forget when adding new ones.

Default unknown locations to excluded. The frontend treated unknown locations as US. The backend does the opposite: unmatched strings get excluded. The tradeoff is that a small number of US jobs with unusual formatting might get filtered, but the alternative is the exact problem that caused the Microsoft incident.

Always-send eval email. The report sends even when everything is fine. If the email stops arriving, I know the cron itself is broken. A report that only sends on failure gives no signal when the monitoring fails silently.

What I Learned

UI filtering is not data filtering. A filter that only exists in the presentation layer protects one view while leaving every other channel exposed. Data quality rules belong at the source, before storage.

Negative patterns are as important as positive ones. Checking only for US patterns would miss a location that contains no US keywords but also is not in any US list. Explicit non-US patterns catch these definitively.

A daily health report changes how you operate. The difference is between reactive firefighting and proactive quality management. Even "all checks passed" across the catalog is reassuring.

Phase 17: Dropping The Fragile Browser

Puppeteer elimination and catalog expansion

Goal: Recover 10 broken companies, eliminate the browser dependency where possible and expand the catalog from 55 to 126 companies.

What I Built

I woke up to an admin email showing 10 companies failing with identical browser crash errors. All used the headless-browser path. The root cause: the Docker base image that bundles the browser had silently updated to a broken version. Every browser-dependent company broke at once.

Instead of just pinning the image, I treated this as a chance to eliminate the browser dependency where possible. For each of the 10, I researched what ATS they actually used by probing known API endpoints. Several turned out to be on platforms I already supported. I built four new scraper types (an iCIMS REST scraper, an HTML-fragment parser, an Amazon Jobs scraper and an Oracle HCM scraper). Seven of the original ten now use API-based scraping. For the remaining three that genuinely require a browser, I pinned the Docker image to a known-good version.

With the infrastructure proven, I expanded the catalog from 55 to 126 companies, researching each one's ATS before adding it. 118 of 126 use pure API scrapers; only 8 need the browser. I also ran a full security and correctness audit that caught 16 issues including a critical blocklist gap.

Key Decisions

Research each company's ATS instead of fixing the browser path. The easy fix was to pin the image and move on. But the browser path is inherently fragile: heavy, slow and breaks when the base image updates. Every company I moved to a direct API call became permanently more reliable. The tradeoff is development time now versus ongoing maintenance time forever.

Accept that some companies genuinely need the browser. A few run sites with no clean API path. For those, the browser remains necessary, but I pinned the image rather than tracking :latest.

Use keyword filtering at the API level where possible. One company's ATS instance has 26,000+ listings; the API's keyword parameter narrows it to a handful. But different instances handle parameters differently, so the scraper uses the filter where it works and falls back gracefully when it does not.

What I Learned

The label "generic" was hiding opportunity. Ten companies were marked "generic," meaning "we do not know the ATS, so use the browser." When I investigated, seven had perfectly usable APIs. "Generic" was not a platform assessment, it was a lack of investigation.

Docker :latest is a landmine. The image updated silently, and because I pulled :latest, every deploy got the broken version. Pinning to a specific tag makes updates intentional. This applies to any base image.

Enterprise ATS platforms have hidden APIs. Documentation aimed at enterprise customers often hides a clean endpoint that returns structured job data. This pattern works across many sites on the same platform.

Phase 18: A Recovery System, Not A One-Off Patch

the self-healing scraper layer

Goal: Stop manually investigating every time a company silently breaks. Build a recovery system that handles platform migrations, dead APIs and lost access automatically.

What I Built

The session opened with one company failing. Their public board, which my scraper had hit for months, returned 404 on every endpoint. The board had been deleted, not renamed, and rebuilt as a custom single-page app behind an internal, access-controlled API, so there was no clean public path to their listings anymore. No clean fix.

Instead of fighting that one company, I built a generalized recovery system. The cron loop now has three tiers when a company returns zero jobs. Tier one is the configured scraper. Tier two is the existing platform auto-detection. Tier three is new: a generic fallback that loads the careers page, watches the structured responses the page itself loads and extracts job listings from any response that looks like a list of jobs. This catches companies that move to custom data sources the system has never seen.

If all three tiers fail seven days in a row, the system auto-disables the company and stops trying. Every Monday, the cron retries each auto-disabled company once; a successful probe re-enables it automatically. The admin email gets new sections for "re-enabled," "fallback recovered," and "auto-disabled."

Key Decisions

Build the recovery system, not the one-off fix. The instinct was to keep digging at the single broken company. I caught myself an hour in and reframed: if I am fighting one company this hard, the next ten will eat the rest of my month. The right scope was a system that turns "manual investigation every two weeks" into "system handles it, emails me only when it cannot."

Auto-disable threshold of seven days, not three. Some companies legitimately have zero PM roles for a week. Three days would auto-disable real companies. Seven is conservative enough that real "no roles" states recover first, but short enough that broken scrapers do not pollute reports for weeks. The Monday probe means even a disabled company gets one retry per week.

Return empty arrays instead of throwing exceptions. Tier-1 scrapers that throw bypass tiers two and three because the catch block fires immediately. So custom scrapers now return an empty array for known failure cases and reserve exceptions for genuinely unexpected errors. The contract for "I tried and got nothing" has to differ from "something is so broken I cannot continue."

What I Learned

Structured data sources are far more reliable than rendering a page in a browser. Some sites simply are not reliably accessible from a server, and the durable answer is to detect that and degrade gracefully rather than keep fighting it. A clean structured feed beats a browser every time it exists.

The "this one company" trap. Every break tempts a one-off fix. After about the third, you have a graveyard of bespoke scrapers and still get woken every other week. The right move is to recognize the pattern early and build the recovery system instead.

Tiered recovery is graceful degradation. Each tier knows less about the specific company but works for more companies. Failing forward through the tiers means the system can usually find at least some answer. Good engineering rarely produces one perfect answer; it produces a chain of progressively-less-confident answers and trusts the most confident one that succeeds.

Phase 19: Less Noise, Tighter Security

Goal: Trust the self-healing system to run quietly. Verify the security claims in our docs match what the code does. Surface new vulnerabilities every week instead of hoping someone remembers to audit.

What I Built

The self-healing layer was working but I was getting three admin emails a day: a scrape report, a quality evaluation and a batch-failure alert. Most of it told me about successes I did not need. I consolidated the three into one digest that fires only when something needs attention: a scrape that failed, a company trending toward auto-disable, a subscribed company that dropped to zero, a delivery batch failure. On Mondays it picks up a weekly digest section. Most days now produce zero admin emails.

While simplifying, I noticed the "fallback recovered jobs" line was firing for thirteen companies a day. Investigation showed most had perfectly functional boards with fifty or two hundred jobs, none of which were PM roles. The primary scraper correctly returned zero, the system fell through to the fallback, the fallback returned the same jobs from a different URL, and the PM filter eliminated everything again. Wasteful duplicate work, not recovery. I added a pre-filter job count, so the fallback only triggers when the source actually returned nothing. Daily fallback runs dropped from thirteen to three.

I also made the fallback useful when it does run: it now returns the URL it found jobs at, and a function maps known URL patterns back to a platform config. When the URL is recognized and differs from the database, the system auto-updates the company's config so the next run goes through the API directly.

Then I ran a full security audit with three parallel review agents, each scoped to one surface: the authentication flow, data isolation between users and infrastructure. They surfaced eleven distinct issues. The most embarrassing was a cookie-security mismatch: a setting added during a refactor quietly contradicted both the docs and a server-side bridge route built specifically on the original assumption. The audit also found a read endpoint that returned company data to non-subscribers, a shared-secret comparison vulnerable to timing attacks and a dependency with a published CVE in its rate-limiting logic. I shipped fixes for all eleven in one commit. To prevent future drift, the Monday email now audits backend dependencies and shows new and resolved vulnerabilities week over week.

Key Decisions

One email channel, not three. Splitting alerts by category feels organized but makes the reader triage which to open first. Folding everything into one channel that fires only when something needs attention is harder to ignore because you know it always means something.

Three specialized review agents instead of one general pass. Each agent read its target files line by line. A generalist pass covers more ground at less depth and would have missed the subtle issue where one read endpoint forgot a subscription check while every other route had one.

Ship the runtime-safe fixes in one commit, defer the runtime-risky one. Ten of eleven were code-only changes I could verify locally. One was a CSP tightening that needs runtime testing because monitoring uses inline scripts. I shipped the verified fixes and logged the deferred one.

What I Learned

A self-healing system that emails you about every success trains you to ignore the channel. By the time something needs attention, you are habituated to skip it. Successful auto-recoveries belong in a database and a weekly review, not push notifications.

Documentation drifts from code during refactors, and audits exist because of that. Someone added a one-line override during a refactor; it was never reverted; the docs kept claiming the original behavior; from the outside everything looked right. Audits compare what you think is happening against what actually is, because no individual change ever sets out to lie.

Specialized matters more than thorough. Where you look is more valuable than how widely you look. Twenty surface checks miss what one deep check uncovers.

Build the monitoring before you need it. The weekly dependency audit runs in the same cron, costs nothing, and catches the next unknown CVE months earlier than the next manual audit.

5. Productionizing and Scale

Turning a working app into an operation: AI helpers, gated deploys, a bigger catalog and a job-first experience.

Phase 20: A Team Of Specialized AI Helpers

specialized AI agents and PR-gated deploys

Goal: Move from "ask Claude to do everything" to a roster of specialized agents, and put a deploy gate in place so a bad change cannot land on users without me seeing a preview first.

What I Built

For about a year, I opened a chat with Claude Code, described what I wanted, then pushed to main. That worked but had two problems. Every task started from zero context, and the same autonomy that made me fast made me nervous: one bad change going straight to production was one careless decision away.

I built a portfolio of thirteen specialized subagents that live in the repo, each a markdown file with focused instructions for a recurring job. Five are custom and encode project-specific knowledge no public agent could have. The other eight were forked from public collections after evaluating around seven hundred agents across four open-source repos, then rewritten to keep only the load-bearing patterns and remove vendor name-drops, marketing taglines and roleplay headers.

Then the deploy gate: branch protection so main only receives code through a pull request, plus preview environments so every PR gets its own working sandbox URL. The workflow now is that I describe a task, Claude routes it to the right specialist, the agent proposes a patch, I review it, Claude opens a PR, preview deploys spin up, I click around, and only then do I merge. The merge button is the only step I cannot delegate.

Key Decisions

Build custom for project knowledge, borrow for generic roles. The reusable ones were generic roles any web app needs. The ones I built encode knowledge no public agent could have: the custom-host blocklist, the three-tier recovery model, the proven audit pattern. I split the portfolio eight borrowed and five custom.

PR-gated deploys instead of direct push to main. The tradeoff is one merge click per change. The benefit is that no change reaches users without me seeing a working preview, and a bad change closes with one click instead of a revert.

Agents propose, the main thread disposes. Every specialist is restricted from running git commands. They return diagnoses and patches as markdown; the main thread handles all git operations after I approve. This keeps risky autonomy where I have visibility.

What I Learned

Generic public agents are surprisingly bloated. Even the highest-quality public agents averaged around fifty to seventy percent bloat for my use case: vendor name-drops, aspirational checklists and "you are a ten-year veteran" roleplay headers. Borrow the bones, throw out the marketing.

Predictable output contracts matter more than the underlying model. Once each agent has a predictable shape, reviewing its output is a thirty-second scan instead of reading prose.

The framework is only as good as the trigger. Owning thirteen agents is useless if I keep doing every task in the main thread out of habit. The win comes from internalizing the routing.

Phase 21: Catching The Silent Zeros

silent zero failures and three new ATS platforms in one night

Goal: Close the silent-zero failure class and recover the catalog companies that had been quietly returning nothing.

A look at the catalog showed about ninety of two hundred and twenty companies sitting at zero PM jobs. Some genuinely were not hiring; some had been, but the scraper had silently failed. The structural problem was that a scrape returning zero looked identical in our logs to a correct scrape for a company with no open roles. Both showed status success. Both produced quality scores. Neither sent an alert. We had built a system that confidently reported nothing while losing real listings.

What I Built

I tackled this in four phased pull requests in one evening, each independently reviewable.

The first added the safety net. Two new columns track verification state: one flips to true the moment a scraper returns more than zero PM jobs (proof of life from the source), the other flips to true only when I manually confirm a zero is legitimate. The daily email got a new section listing every company at zero without sign-off, sorted by subscriber count. The same PR fixed a long-running Ashby bug where the scraper filtered jobs by team name; modern companies put product managers inside teams named after products, not "Product Management," so the filter was silently dropping real PM roles at about seventeen companies. Switched to title-keyword filtering.

The second built a real scraper for Apple, whose careers site is a single-page app with no useful HTML. The fix was Apple's own internal API: a token endpoint followed by a paginated search endpoint. The payload required a specific field that, if missing, made the API silently return zero records, exactly the failure mode my old monitoring would have missed. The same PR fixed an EA scraper dropping ten jobs per page from a wrong selector, and added hostname guards for two companies that block server-side requests at the edge, routing them to the fallback tier.

The third expanded the keyword list to match the function name "product management" in addition to the role name "product manager." This sounds small but recovered American Express from one job to dozens; several companies put the function name in their titles, like "Senior Manager - Product Management."

The fourth built scrapers for two companies I had deferred: Shopify (which embeds another vendor's board, returning null from the obvious API but server-rendering jobs into a streaming payload) and eBay (on an ATS I had not supported, where the data sits in an inline object on the page). I documented an honest limitation: that ATS only server-renders ten jobs per page, so we capture a first-page sample, and I set the scanned-count to the API's reported total so the self-healing tier knows the source is live.

Between the four PRs, I deleted four companies, each at zero PMs for months with zero subscribers and no clear path back.

Key Decisions

Phase the work into four pull requests, not one. Each had an independent verification path and risk profile. Smaller PRs let the change reviewer give focused feedback, and one revert button per phase is safer than one mega-revert.

Trust live tests over cited prior art when they conflict. Prior-art research pointed at a published Apple scraper using specific endpoints. The agent had reverse-engineered different ones. I tested both: the prior-art endpoints 404'd; the agent's returned valid data once I added a mandatory field the prior art omitted. Cite prior art as a hypothesis, always verify with a live test before committing.

Hard-delete the unrecoverables instead of suppressing them forever. None of the four were salvageable, none had subscribers. Marking them legit-zero forever lets them sit invisibly as a slow-growing pile of "things we gave up on." Deleting keeps the catalog honest, and re-adding is straightforward if circumstances change.

What I Learned

The fix for a silent-failure class is not a smarter scraper, it is a different definition of success. Our monitoring defined success as "the scraper returned without throwing," which is structurally incapable of catching a scraper that returns the wrong answer cleanly. The fix was verification columns that force every zero through a decision.

One pre-filter step can be more wrong than the rest of the code combined. The Ashby team-name filter was a reasonable design that encoded an assumption about how companies structure teams, an assumption that stopped being true years ago. Look hard at the assumptions baked into filter logic; that is where silent bugs live.

The catalog hygiene step is part of recovery, not a separate cleanup. I deleted four companies because the audit forced a per-company decision. Expect the catalog to oscillate as scrapers break and the email forces "fix or remove" calls.

Drilling deeper on the "why"

The pivotal move in this phase is conceptual, not technical: I stopped defining success as "the code did not crash" and started defining it as "the code produced evidence it works." Those sound similar and are worlds apart. The first definition can never catch a clean wrong answer, because a clean wrong answer does not crash. By adding a column that records "this source has, at least once, produced a real PM job," I gave the system a notion of proof-of-life that is independent of whether today's run happened to throw. Everything else, the daily zero-list, the manual sign-off, the auto-flip logic that comes one phase later, is built on that single shift in what counts as success.

Alternative considered I could have kept investigating the ninety zeros by hand, company by company, which is what the old workflow implied. With ninety companies that is days of spot-checking, and it scales linearly with every future zero. I chose to spend the evening building the verification machinery instead, accepting a noisy daily email during the triage backlog as the intended cost. The hand-investigation finds today's ninety; the machinery finds today's ninety and every silent zero from here forward, and turns each one into a forced "is this real?" decision rather than a thing I have to remember to go look for.
Edge cases that bit The sharpest boundary case was Apple's API silently returning zero records when one required field was missing from the request. The endpoint did not error, it just answered "no jobs," which is the single most dangerous shape of failure in this whole product because it is indistinguishable from a real empty result. The other live one was the function-versus-role naming gap: companies titling a job "Senior Manager, Product Management" rather than "Product Manager" fell entirely through a keyword list that only looked for the role name, so a company with dozens of PM roles read as having one. Both are cases where the data was right there and a single narrow assumption hid it.
What this unlocked This phase recovered a large slice of the catalog that had been quietly dead, and it converted "silent zero" from an invisible class of bug into a visible daily decision. The verification columns it introduced are the foundation the next phase builds on to remove me from the loop entirely, by teaching the system to make that "is this zero real?" decision on its own.

Phase 22: Growing The Catalog The Right Way

targeted catalog growth after the Common Crawl detour

Goal: Scale the catalog from 216 toward 500-1000 companies, but only with companies the audience actually job-hunts for.

After the zero-jobs audit closed, the catalog had hit a steady-state of about 216 companies, mostly tech. Two paths surfaced. The first was broad: a pipeline that discovers any company with a public board on the big platforms. The second was narrow: pick verticals the audience cares about and add recognizable companies by hand. I tried both, and the difference between them is the entire game.

What I Built

The broad path was a Common Crawl harvest. Common Crawl is a free public web index; it lets you query for every page matching a URL pattern. The script pulls candidate company slugs, validates each by calling the ATS's own public API to confirm the board exists and has jobs, then emits the survivors as ready-to-apply SQL. The run found 1,352 working companies; after de-duping against my catalog, 1,328 truly new candidates.

I looked at the top fifty by job count and almost none were companies I recognized. Below that the list was home-health-care providers, autism therapy networks, apparel brands, regional architecture practices. The pipeline worked perfectly. The audience-fit was wrong.

So I shelved the pipeline (kept the code) and switched to targeted vertical batches: four days, four batches of recognizable banks, biotechs and consultancies. Net: twenty-nine high-recognition companies added, five entirely new ATS platforms supported and a real strategic shift from "discover broadly" to "curate carefully."

Key Decisions

Take Common Crawl seriously enough to build the pipeline, then ship nothing from it. When you have spent time building infrastructure, the temptation is to use it. I almost shipped the top fifty just to validate the build. Then I read the names. The cost of polluting the catalog with hundreds of unknown companies was higher than the cost of admitting the pipeline was the wrong shape. I kept the code and walked away from the output.

Build the new ATS scraper as a generic platform scraper, not a one-off. The first company was the trigger but the design target was "every large enterprise on this platform." That paid off two days later when another company needed it and the same code worked with a one-line change.

Trust the consulting industry's choice to be hard to scrape. The big consultancies are on a login-gated, bot-resistant platform with no public path. Each would need an architectural change for roles that are mostly consulting titles anyway. The right call was to stop trying and remove them from consideration permanently.

Verify scraper yield with a simulation before declaring a win. After shipping five custom scrapers, I ran a verification agent that fetched live data, applied my real filters and reported the actual PM count. Two of five returned zero US PMs despite working perfectly: one hires consulting titles, the other is concentrated in Europe. If I had skipped verification, both would have sat in the admin email forever as silent zeros.

What I Learned

Breadth and depth are different problems, and scaling along the wrong dimension is worse than not scaling. Every growth lever has an implied audience. If the lever does not match who you are growing for, more of it makes the product worse.

Industry-wide ATS patterns are real and worth pattern-matching. Whole industries cluster on one platform. Once I recognized these patterns I could pre-predict the scraper outcome, cutting per-company research time roughly in half.

Honest verification of yield matters more than "endpoint returns 200." I shipped five scrapers, all returned 200, all parsed correctly. Verification found two delivered zero user value. The fifteen extra minutes saves weeks of email noise.

Parallel agents collapse multi-day work into hours when the work is independent. Twelve parallel investigations across this period ran in about 25 to 30 minutes per batch; done sequentially it would have been a full week.

Drilling deeper on the "why"

The real lesson of this phase is that "more companies" was never the goal, it was a proxy for "a catalog that feels like a real database to the person using it," and the proxy and the goal pointed in opposite directions. A thousand companies nobody recognizes makes the catalog look bigger and feel emptier, because a job-hunter scrolling past names they have never heard of concludes the tool does not cover the companies they care about. Twenty-nine recognizable names do more for that feeling than 1,328 unknown ones. Growth measured by row count was optimizing the proxy; growth measured by recognizability was optimizing the goal.

Alternative considered The path the sunk cost was begging me to take was to ship the Common Crawl output, at least the top fifty by job count, since the pipeline already worked and the SQL was already generated. I rejected it because a catalog is judged by its worst visible entries as much as its best, and a feed full of regional staffing firms and therapy networks signals "this tool is not for me" to exactly the senior-PM audience I built it for. I chose to keep the pipeline as dormant tooling for some future use case and hand-curate instead, trading a big number for a catalog whose every entry earns its place.
Edge cases that bit The instructive boundary case was the pair of scrapers that passed every technical check and still delivered nothing. Both returned a 200, both parsed cleanly, both looked like wins, and both produced zero US PM roles once my actual filters ran, one because the company hires consulting rather than product titles, the other because its openings are almost entirely in another region. "The endpoint works" and "this is useful to my user" are different claims, and only a full simulation against live data and real filters tells them apart. Skipping that step would have planted two new silent zeros in the exact catalog I had just spent a phase cleaning of them.
What this unlocked Beyond the twenty-nine names, this phase produced a repeatable growth motion: recognize the industry's platform, spin up parallel agents to research, build generic scrapers that unlock many tenants at once, and verify real yield before counting any of them. That motion is what makes the catalog-doubling in Phase 30 tractable instead of a slog.

Phase 23: A Job-First Feed

comp-filterable, and the email-disappearance bug

Goal: Turn the catalog into something a real job-hunter would actually browse, fix the slow accumulation of email-quality issues and find out why I stopped getting my own daily email.

The catalog had grown to 244 companies. Browsing it as a list of cards felt right at 50 and wrong at 250. A real job-hunter wants to see jobs across all companies sorted by what matters to them. In parallel, the daily email was getting noisier. Then on the last day of the sprint, I noticed I had not received my own daily alert. The provider showed zero record of trying to send to me. That kicked off the most consequential bug of the sprint.

What I Built

A parallel job-first feed route. A flat table with one row per job: Company, Title, Location, Level, Comp, Posted, Track. Filters across the top: search, sort, level, industry, company, region+city, minimum comp, include-closed. Logos use a three-step fallback chain. The Track button is auth-aware. I built it as a parallel route rather than swapping the existing home: the marketing landing has a known conversion rate, so I can compare live for a week, then swap if it is better. Cheap reversibility.

Email quality overhaul, several small compounding fixes: a zombie-return fix (jobs that flicker off and back were being flagged "new"), a fourteen-day return rule, recommendation rotation via a history table (replacing a broken day-of-month scheme the user caught immediately), a firehose sort (companies with ten-plus new jobs go to the bottom) and a per-company seniority threshold so junior FAANG titles do not spam senior PMs.

An admin email-status diagnostic that proxies the provider's search API, so I can answer "did user X get email Y" without writing per-recipient send metadata to my own database. I deliberately wanted those emails to live in the provider, not my database.

The pagination bug. The cron loop was not iterating over my user record. Root cause: the function that lists all users defaults to a page size of fifty. We had crossed fifty total users the day before, so the oldest users, including me, were silently paginated out of the daily run. A one-line unblock went out within minutes; a proper paginated helper followed, and nobody calls the raw function directly again.

Key Decisions

Build the feed as a parallel route, not a swap. Swapping a converting landing page for an unproven surface is a one-way door with a measurable cost. Building in parallel makes the eventual swap a one-line change with a clean rollback.

Manually seed FAANG seniority thresholds instead of waiting for automation. The long-term answer reads each company's early-tier pay automatically, but that is a backlogged build. Manual seeding for the six FAANG companies took five minutes and filtered the noise now.

Query the provider for ground truth instead of building our own send log. A send-log table would have put user emails in our database forever. Proxying the provider's API is zero new tables, zero new PII surface and a strictly more powerful debugging interface.

Patch the pagination bug at every call-site immediately, then refactor to a helper. Patching got the email back within an hour; the refactor an hour later guards against the next call-site repeating it.

What I Learned

Default values in SDKs are landmines when the default looks reasonable until you cross an invisible boundary. A page size of fifty worked for the first fifty days. The fifty-first user was the cliff. Any paginated call now gets explicit handling, never the default.

When an external service handles delivery, query that service for ground truth. Their log is more accurate than ours will ever be. A thin proxy endpoint surfaces it through our auth surface with no PII written down.

Cheap reversibility beats perfect-on-first-try. I shipped seventeen changes in two days because most were cheap to roll back. The version that ships is not required to be the version that lasts. It is just required to be safe to throw away.

Our query layer has silent failure modes that look like working features. Sorting by a related table quietly does nothing; a filter breaks if the value contains a comma; a case-insensitive match turned a state abbreviation into a substring match on another state. Each was a one or two line fix once I knew about it; the cost was the half-day spent finding the failure mode.

Phase 24: Turning The Data Into Content

the weekly LinkedIn digest

Goal: Use the catalog data to drive a weekly LinkedIn post, both for the awareness flywheel and to give my audience a reason to come back every Friday.

The catalog had matured: 244 companies producing 500+ new PM roles every week, with industry concentration and AI-hiring cross-cuts that genuinely surprised me. None of that signal was visible outside the product. There is a person on LinkedIn I follow who does a weekly "PM roles by level" post; it works. I did not want to copy the format because my data has cross-cuts theirs does not. The differentiation was already in the data; the job was packaging it.

What I Built

A weekly digest email that lands every Friday with a copy-paste-ready LinkedIn post. The structure is now editorially locked: an intro line, the week's total, an industry-concentration lead, a top-ten-by-volume list, a spotlight company, an AI-PM section and a reader-question close. The close serves double duty: it is the standard engagement driver, and it crowdsources future content angles.

Auto-trigger inside the existing daily cron, gated to Fridays only. No new schedule. This is now the canonical way to add any weekly task: hang it inside the daily job, gate by day of week, never stand up a new scheduler entry.

A voice style guide and calibration samples that future AI sessions must read before drafting anything in my voice. The AI violated my anti-slop list within ten seconds of saving the memory pointing at the file. Saving the pointer did not change behavior; only reading the actual file content does.

Key Decisions

Single weekly post with multiple sections vs. rotating one card per week. The initial proposal was a five-week rotation. I rejected it: a single richer post is a better artifact, lets me audible to the strongest lead each week, and reads like a trend dashboard rather than a recurring listicle.

Comp as supporting data, not lead. Comp data exists for about a third of roles. Leading with it risks implying we have comp for everyone and putting specific companies in a public "low comp" position. It moved into the email body as a verification table, not into the post.

Embed in the daily job vs. a new Friday schedule. Every extra scheduler entry is one more surface to manage. One entry point with day-of-week gates is cleaner.

Voice as reference-pointer, not inline memory. The files are large and edited over time; a small memory entry says "READ these first" and lets the source of truth stay in the voice docs.

What I Learned

Proprietary data should be packaged in cuts that are not obvious to outsiders. Anyone can post "highest-paying PM roles" from a public salary database. Almost nobody can post the industry-concentration cross-cut without the underlying watchlist. The data is the moat; the content is the visible artifact of it.

AI-assisted content in someone else's voice needs structural guardrails, not stylistic hints. Telling the assistant "match my voice" does not work, and neither does a memory that says "see the voice doc." The fix is making the voice files required reading at draft time. The honor system does not work for content tone any more than for security.

The cron-day-of-week pattern is reusable infrastructure. Once it was clearly a pattern, I wrote it down: hang it inside the daily job with a day-of-week gate. Never add a new scheduler entry.

Phase 25: The Auth Incident

Goal (in retrospect): Brand the auth emails to match the rest of the product.

What it actually became: an incident, a deep cookie-bug discovery, a feedback-workflow migration and a three-layer defense-in-depth infrastructure that did not exist a week earlier.

What Happened

I had just shipped a brand-styled redesign of the bug/feedback admin emails. The same visual language was the right move for the user-facing auth emails too: Confirm Signup and Magic Link. Those templates live outside the repo, in the auth provider's dashboard. I re-deployed both via a one-shot script. Verified the deploy looked right in a browser preview. Done.

Two days later, two users emailed me: "I am not getting the login email." I queried the user table. A batch of users were locked out. One user's record showed she had clicked the link (account marked confirmed) but never been logged in (no session). Classic "confirmed but no session": she got past the click, but no session cookie ever reached her browser.

I put the incident-triage agent on the broken templates. It found the smoking gun within minutes. My deployed templates used the provider's default link variable. That variable builds links pointing at the provider's own hosted verify page, not at our app's confirm route. The hosted page validates the token (so the account gets marked confirmed) and then bounces back to our site, but the session cookie is never set, because our own confirm route never runs. The documented format, sitting in the auth sidecar I should have read before deploying, was the opposite: send people to our own app's confirm route carrying a one-time token. The sidecar's enforcement hook is folder-based; my deploy script lived in a different folder, so the hook never fired. I trusted a guardrail to catch me in the one spot it could not reach.

The Deeper Bug

Even after re-deploying with the correct format, some users still hit "confirmed but no session." I put a debugger agent on the confirm route. The finding: our redirect was building a fresh response that did not carry the session cookie the verification step had just written. The browser got the redirect with an empty cookie header, so the session was gone before the next page loaded. This bug had been there since the confirm route was first built. The Linear issue tracking it was titled "Magic link fix, non-Gmail, Safari mobile." That title was a misdiagnosis. The bug is universal; Safari Mobile just hits it more reliably because its session restoration is slower and loses the race more often.

What I Built

Layer one, fix everything broken: repointed the templates at our own confirm route; fixed the cookie-drop in both login routes (copy the session cookie onto every redirect); built an admin endpoint to send a fresh magic link to a specific existing user, recovering the stuck batch.

Layer two, three independent defenses: a CI auth-template check on every auth PR plus nightly that fails the build if the wrong link variable appears; a daily code audit that scans the last day's changes for four known risk patterns and emails me only on a find; and auth-funnel analytics (email sent, link clicked, signed in, failed) so a drop-off between "clicked" and "signed in" would have paged me about six hours after the deploy.

Layer three, workflow shift: migrated user feedback off our database tables onto a dedicated Linear team, so triage is conversational and the admin email carries the issue URL inline.

Key Decisions

Three layers of defense, not one. A single "remember to do X" rule has a 100% failure mode. Three independent layers, PR-time CI, a 6-hour analytics alert and a 24-hour daily audit, each catch this class independently. The time-to-detection math: 48 hours became about 10 minutes worst case.

Observability in observability tools, not the application database. Errors to the error tracker, funnels to the analytics tool. The application database is for application data.

Incident response and prevention in parallel. The temptation is to firefight, then breathe. The right move is to firefight and build the prevention layer simultaneously, while the failure mode is fresh.

What I Learned

Cosmetic tasks are not cosmetic when they touch load-bearing config. Framing the template change as "design polish" was the original sin. The fix is not "read sidecars next time," that is an honor system; it is a CI check that fails the build if the deployed template does not match the documented format.

A preview that does not simulate the actual user action is not a preview. I generated standalone HTML previews that looked great but could not simulate the link click. A real preview goes through the actual send-and-click loop.

Mislabeled issues hide root causes for months. The ticket title encoded the symptom, not the broken artifact. Title issues by the architectural thing that is broken, not the user-visible failure.

SDK defaults are landmines, again. Combined with Phase 23's pagination default, this is a pattern: every SDK has invisible cliffs behind reasonable-looking defaults. Never use an SDK default for anything load-bearing.

Drilling deeper on the "why"

The thing worth sitting with is why a two-character difference, the default link variable versus the documented one, could lock out a batch of users while every check I ran said green. The deploy looked right because the preview tested the only thing a static preview can test, layout, and was structurally blind to the one thing that mattered, what the link actually does when clicked. The deeper bug compounded it: even with the correct link, our own redirect threw away the session cookie. So there were two independent failures stacked, and the reason neither surfaced is that nothing in my workflow exercised the full send-click-session loop end to end. The fix could not be "be more careful," because I was being as careful as the tools allowed. It had to be a new check that performs the user's actual action, which is exactly what the CI template check and the funnel alert do.

Alternative considered The cheap response was to fix the template, recover the users and add a note-to-self to read the auth docs before touching templates next time. That is a one-hour fix and it closes the immediate incident. I rejected it because a "remember to do X" rule for a task I perform regularly has, over enough repetitions, a 100% failure rate, and this incident was itself the proof: the folder-based guardrail that was supposed to catch exactly this could not reach the folder my deploy script lived in. So I built three automated layers that catch the class without depending on my memory, and I accepted the extra build time as the cost of converting a 48-hour blind spot into a roughly 10-minute one.
Edge cases that bit The defining boundary case is "confirmed but no session," a user who is, from the database's point of view, a successfully verified account, and from her own point of view, completely locked out. Two contradictory truths about the same person, produced by a token that got validated on someone else's hosted page while the cookie that proves it never reached her browser. Safari Mobile made it worse and noisier precisely because its slower session restoration loses a race more often, which is why the issue got mislabeled as a Safari bug when it was universal. And the guardrail edge case is the cruelest: a folder-scoped hook that protects the auth folder is silent the moment the dangerous action is performed from a script in a different folder.
What this unlocked The incident is what turned my auth flow from "tested by hand and hoped" into "watched by three independent alarms." It also produced the admin recover-a-user endpoint, the move to a real feedback workflow, and the principle, repeated in later phases, that prevention is cheapest to build in the hour you are recovering from the failure, not a month later when the urgency is gone.

Phase 26: Letting The System Decide For Itself

removing the admin from the operational loop

Goal: Stop the system from asking me to confirm things it could decide for itself.

Two streams of operational noise hit the same morning, and the shape was the same in both. The analytics tool sent me six emails about an auth-conversion drop that was a phantom; the alert math was just unstable. And my daily digest asked me to manually mark twenty-eight companies as legitimate zeros before it would stop bothering me. Both systems were doing exactly what I asked. Both were asking the human to do work the system already had enough information to do itself.

What I Built

Replaced the unstable ratio alert with a flat volume alert. The old alert computed successful signins divided by emails-sent over a rolling 6-hour window. On low traffic that is statistical noise: a user who requests a link at 9am and clicks at 4pm lands in the success count with no matching email-sent in the same lookback, so the ratio swung wildly on event timing, not on whether auth worked. I replaced it with a chart that counts successful signins over 24 hours and alerts only if that drops below 1. That catches catastrophic auth failure without misfiring on timing.

Added missing instrumentation to the same-browser auth path, which had been firing zero analytics events, so any signin through it was invisible and the funnel undercounted.

Ripped out the manual unverified-zeros triage entirely, replaced with three automatic rules: auto-verify a zero after seven days from a scraper that has proven itself; auto-disable a never-proven scraper after fourteen days; and auto-flip back the moment any scrape returns even one PM. There is no permanently-silenced state; every scrape re-evaluates from scratch. I bootstrapped all 28 currently-zero companies to silenced and deleted the whole email section and its dead code.

Key Decisions

Volume alert vs. anomaly detection on the ratio. Anomaly detection adapts its baseline to recent noise, which means it adapts to zero traffic as baseline. A flat volume floor catches a complete outage even when traffic is low. Coarser measurement, more reliable signal.

Aggressive bootstrap vs. conservative scope. The first cut only silenced proven scrapers, but that matched zero of the 28 actual noise items. The pragmatic version silences all currently-zero rows and trusts the auto-un-silence logic to restore accuracy. Safe because the flag has zero user-facing blast radius.

Three rules in cron vs. a separate background job. The signals these rules act on are all updated by the daily scrape itself, so putting the logic inline evaluates on the freshest state with one scheduled job.

What I Learned

The flag's blast radius determines whether bold defaults are safe. I searched the whole codebase for the flag before going aggressive: read in four backend files, nothing user-facing. That bounded downside is what made the aggressive bootstrap safe. The semantics are only safe to relax if you know what is reading the flag.

Cross-window ratio alerts misfire on low traffic in ways volume alerts do not. Any rate-of-X alert that divides events-in-window-A by events-in-window-B drifts when matching events land in different windows. For low-traffic metrics, a flat volume floor beats a conversion ratio every time.

When the data evolves, the workflow has to evolve with it. The "unverified zeros" email was right when it shipped, then outlived its information environment by months once the system could tell a proven scraper from an unproven one. Every workflow that asks a human to do something the system could do is unpaid tech debt.

Phase 28: The Weekly Post Writes Itself, In My Voice

Goal: Stop opening every week with the same headline, and stop hand-writing the lead.

The weekly digest from Phase 24 had a quiet problem: the headline was hardcoded. Every Friday it announced the same lead, no matter what the week actually looked like, because the lead was a fixed line in the code and only the percentage moved. When I finally looked closely, it was not even true. The "surge" it claimed was an artifact of my bulk-adding a batch of companies that week, whose existing openings landed as "new." I was one paste away from publishing my own data-pipeline quirk to my whole network as a market take.

What I Built

A lead that rotates and writes itself. Each week the system works out a handful of honest, this-week angles (how much of the hiring is AI, who posted the most, how concentrated big tech is, who pays the most, where the jobs are), picks one it has not used recently and has an AI write it as a blunt one-line take in my voice, with the rest offered as alternates. It also makes a branded banner whose art style rotates weekly, so the series looks fresh but unmistakably mine, and emails me the whole kit every Friday to approve before I post.

The voice was the hard part, and it humbled me. I have a written style guide and a file of my real posts, and the drafts still came back as generic AI mush, several times over. The fix was not a better description of how I write. It was feeding the AI the actual files, word for word, on every run, and keeping myself as the last reader. A summary of your voice is not your voice.

What I Learned

Your own data can lie to you in a shape that looks exactly like insight. The headline I had repeated for weeks was a fact about how I grow the catalog, not about the market, and I only caught it because the repetition annoyed me. Lead with the cut nobody else can see, but prove it is real before you publish it.

Phase 29: Auditing The Whole System Before A Major Expansion

Goal: before a major expansion, make sure the foundation is sound, and fix the one thing that genuinely scared me.

I was about to expand the system significantly, which opens new ways for things to go wrong. So before building anything new, I stopped and audited the whole system end to end: the scrapers, the security headers, the login flow, whether one user can see another's data and where the app is slow. I asked for a plain report first, not fixes on the fly, so I could read it and decide what mattered.

What I Built

Most of it was healthy. But the audit kept finding the same shape of problem, the same one that had bitten me twice already that week: things that fail silently. A query that quietly returns only the first thousand rows. A safety check that is switched off by a missing setting. A scrape that finds nothing and still reports "success." None of these throw an error. They just quietly do less than they should, and silence looks exactly like health.

Two findings actually scared me. The first: a single capped query had silently dropped a large share of my subscribers from a daily email earlier that week, and nothing caught it but a user. The second: two database tables were technically readable by the public browser key, one of them holding feedback people had submitted.

So I fixed the foundation, each change reviewed independently before it shipped: swept out every remaining query that could silently truncate and added an automated check that blocks new ones; locked down the two tables and added a rule that a new table cannot ship without that lock; and stopped a pointless, legally risky arms race I had been running against a few employers' access controls.

The email fix is the one I cared about most. I did not want one safeguard, I wanted a backup and a backup for the backup. So I built it in layers, each catching a different way the same disaster could happen: the recipient list can no longer be cut short at the source; if a day somehow produces zero emails when there are people to email, it screams before sending anything; after sending, it checks what it meant to send against what it actually sent; and every day it records how many people were eligible and compares that to the recent normal, so a slow erosion gets caught within a day even if every other check passes.

The last piece was about honesty. A handful of big employers block us from reading their careers pages. We had been showing "0 roles" for them, which makes it look like they are not hiring and the product looks broken. Now they show a plain "Scraping blocked" badge that tells the user what is happening and to apply on the company's own site, and you cannot add them to your tracker since we cannot actually track them. Rather than keep fighting their access controls, I decided to license that data from a jobs-data provider instead, so those companies will come back with real listings soon.

What I Learned

Audit before you expand the surface, because every new feature widens the surface you can get hurt on. And the throughline of the whole phase, the same lesson the weekly headline taught me, is that the most dangerous failures are the quiet ones. A loud failure gets fixed in an hour. A silent one runs for months looking like everything is fine. Defense-in-depth is not paranoia, it is putting a different alarm at each altitude so silence cannot pass for health.

Drilling deeper on the "why"

The reason I built the email protection as four layers instead of one is that the original failure was not caused by the absence of a check, it was caused by a single check being the only thing standing between a silent truncation and every subscriber's inbox. One safeguard has one failure mode, and when it fails it fails completely and quietly. Four safeguards at four different altitudes, at the source, before send, after send and against the historical baseline, fail independently. For the same disaster to slip through, all four have to miss it in the same run, which is enormously less likely than one missing it. The insight is that for a silent failure, redundancy is not over-engineering, it is the only thing that converts "nobody noticed for a week" into "caught within a day."

Alternative considered The natural reaction to the truncation incident was to fix that one query, cap-proof it and move on. It is the proportionate response to a single bug. I chose instead to treat the truncation as one instance of a class, silent partial failure, and to build both a codebase-wide sweep for every other query that could do the same thing and an automated check that blocks new ones from being written. The single fix protects against the bug I already found; the class-level response protects against the next one I have not. Given that this exact failure shape had already bitten me twice in a week, betting on "I will catch the next one by hand" was the option the evidence had already disproven.
Edge cases that bit The defining silent-failure trio of this phase: a query that returns only its first thousand rows with no error, so a daily email quietly skips a large slice of subscribers; a safety check disabled not by code but by a missing configuration value, which no code review can see because the code is correct and the setting is simply absent; and a scrape that finds nothing yet reports success. Plus the honesty edge case, the blocked employers showing "0 roles," which is technically not a crash but is a lie of omission that erodes trust more than an outage would, because it makes a working product look broken and a hiring company look idle.
What this unlocked This audit is what made the big catalog expansion in Phase 30 safe to attempt: I widened the surface only after hardening the foundation under it. The layered-email pattern and the "new table cannot ship without its access lock" rule became standing invariants, and the honest "Scraping blocked" badge replaced a quiet lie with a clear explanation, which is the same honesty principle the measurement phases push even harder.

Phase 30: Doubling The Catalog And Making "Add" Actually Work

Goal: make the catalog feel like a real database, and make adding a company actually show its jobs.

Two problems had been nagging. The catalog had stalled around 247 companies, mostly the tech names everyone already knows. And worse, adding a company barely worked: when I added one, nothing showed up until the next day's scrape. The reason was a design knot I had tied early without noticing. Scraping a company and emailing everyone about new jobs were the same job, welded together in the daily run. So there was no way to scrape a newly-added company on its own without also firing a daily email to every user.

What I Built

Growth. A popular salary-comparison site publishes a public ranked list of tens of thousands of companies; I used that as a discovery source, then verified each candidate's board actually returned real PM roles before counting it. The catalog roughly doubled, from 247 to 519. To make a bigger catalog usable, I built an "add by category" flow: browse by industry and add a whole group at once, with the tech bucket split into sub-categories like AI, developer tools and security.

Cutting the knot. I split scraping apart from emailing, so the system can scrape any set of companies on demand without sending a single email. Then I wired that into the moment it matters: when you subscribe to a company that has no jobs yet, the system quietly scrapes it in the background and its roles appear within a couple of minutes instead of the next day. The daily run is still the safety net.

The employers we cannot scrape. A handful of big names block us outright, so I built the licensed-data restore this session. It pulls those companies' real openings from the provider and quietly switches them back on, written to turn itself on automatically once the provider's free quota resets, so there is no manual step; until then it sits dormant. An independent review of my own code caught a real bug before it shipped.

What I Learned

A catalog you can add to is worthless if adding does not show results. The thing blocking that was not a missing feature, it was an assumption baked into the architecture a long time ago, that scraping and emailing always happen together. The fix was to question the coupling, not add another button. And growth only counts if it is names people recognize and boards that actually return jobs, not a bigger number for its own sake.

Phase 31: The Night The Daily Job Died Silently

Goal: make it impossible for the daily job to fail without me knowing.

The day after I doubled the catalog, the daily run that scrapes every company and emails everyone their new jobs simply did not happen, and nothing told me. I found out hours later by checking the database by hand. What happened: the bigger catalog made the run take far longer, and while it was still running I shipped an unrelated change, which restarted the server and killed the run partway through. No email went out. No alarm fired.

Why no alarm fired is the part worth sitting with. Every safeguard I had built lived inside the same job and ran at the very end of it: the email-volume check, the status email to myself, all of it sat after the long scrape that died. And a server being killed is not an error the code can catch, it just stops. So a job that never finished looked exactly like a quiet, healthy day. Broken, still running and never started were all indistinguishable from the outside, and the health check stayed green the whole time because a fresh server had taken over fine.

What I Built

I treated "the run finished" as something that has to be proven from outside the run. A watchdog now runs on a different system entirely, on its own schedule, and asks the app one question: did today's run actually complete? If the answer is no, or the app does not even answer, it emails me. The job also writes a heartbeat the moment it starts, so a run that dies halfway leaves a visible "started but never finished" mark instead of nothing. If a deploy kills the run, the shutdown now fires a real alert naming the exact company it died on. I made the run resumable, so a restart picks up where it left off. And I made it an enforced rule that nobody can ship a change during the daily run's window.

While fixing this I found the same disease elsewhere: an email alert I thought was protecting me had been quietly dead for who knows how long, because a setting it needed was never configured, and a code review cannot see a missing setting. So I added a monthly probe that actually sends a test alert and confirms it arrives, a dead-man's-switch for the alarms themselves.

What I Learned

You have to monitor the monitors. Every guardrail I had checked some specific thing, but nothing checked whether the work even ran, or whether the alarms could still go off. The fix for a silent failure is never another check inside the thing that failed. It is a witness standing outside it. The most dangerous outage is the one that looks identical to everything being fine.

Phase 32: Seeing The Funnel, And Catching Myself Guessing

the funnel dashboard

Goal: stop running the product on gut and actually see where people fall off.

For months I had the basic analytics: pageviews, clicks, the usual. But I could not answer the one question that actually matters. Of the people who sign up, how many do the thing the product is for? I was guessing. And when you guess, you build the wrong stuff.

What I Built

So I built a real funnel dashboard. Acquisition, activation, retention, the whole path, a couple dozen views, all from data I was already collecting. And the moment I could see it, the picture flipped on me. I had assumed my problem was getting people in the door. Wrong. Acquisition was fine. The leak was activation: roughly 40% of the people who sign up never track a single company, and almost nobody stars a job. The product works once you use it. The hard part is getting someone to that first real action.

Then I went to measure the daily email, which is the core loop of the whole product. Turns out you basically cannot measure email opens anymore. Across 100 sent emails, including the ones to my own inbox that I know I opened, the tracker recorded zero. Modern mail apps block the tracking pixel by default. Clicks I had deliberately turned off, because the same email system sends my login links and I did not want to rewrite those. So I built my own click tracking: every job link in the email now runs through a signed redirect on my server that records the click and then sends you straight to the job. Signed, so nobody can turn it into a tool for pointing my domain at a malicious site.

Here is the part that stung. I had two tickets I was excited about, both promising big job-count recoveries from scraper fixes. One I had written up myself as recovering jobs "across the whole catalog." I built the fix, then actually measured it. Ten jobs. Another ticket claimed 41 jobs for one company. The real number after filtering was eight. The whole point of the dashboard was to stop guessing, and the first thing it proved was that I had been guessing about my own work too, and guessing high. So while I was in there, I built the project's first test harness, so the filter that quietly dropped real jobs twice before cannot do it a third time without a red light.

What I Learned

Measuring changes what you build, and it humbles what you claim. Seeing the funnel told me to fix activation, not chase signups. Seeing the real email numbers told me opens are a fantasy and clicks are the only honest signal. And measuring my own "wins" told me the number is almost always smaller than the ticket says. Build the instrument first. Then believe the instrument, not the ticket and not yourself.

Drilling deeper on the "why"

The uncomfortable insight underneath this phase is that running on gut does not feel like guessing, it feels like knowing. I was confident the problem was acquisition, confident my scraper fixes had recovered large numbers and confident enough to write one of those numbers into a ticket myself. The dashboard did not just give me data, it gave me a way to be proven wrong by my own product, three times in one session. That is the actual value of an instrument: not the chart, but the friction it puts between a confident belief and a shipped decision. Without it I would have spent the next stretch optimizing signups, the one part of the funnel that was already fine.

Alternative considered For email engagement, the conventional choice is to lean on open tracking, which every email provider offers out of the box, and call it the engagement metric. I could not, because the measurement is dead: across 100 sends, including my own opens, the pixel reported zero, since mail apps now block it by default. So rather than report a number I knew was fiction, I built a signed click redirect of my own, which measures a real action instead of a blocked pixel. The cost is that I had to build and secure the redirect myself; the benefit is the only honest engagement signal I have. Reporting opens would have been easier and would have been a lie.
Edge cases that bit The signed part of the redirect is the edge case that mattered. A plain server endpoint that takes a destination URL and forwards the browser to it is an open redirect, the exact vulnerability class I had fixed back in Phase 9, now reintroduced in the name of analytics. If unsigned, anyone could craft a link on my own trusted domain that forwards to a malicious site. Signing every link closes that. The other boundary case was the test harness: the job filter had silently dropped real listings twice across the project's history, a failure with no exception and no symptom until someone noticed missing jobs, which is exactly the kind of quiet regression a test exists to turn loud.
What this unlocked This phase pointed the next stretch of product work at activation, the real leak, instead of acquisition, the imagined one. It established clicks as the product's honest engagement metric for everything that follows, and it added the first automated test to a codebase that had been relying on manual checking, which is the seed of the "trust the instrument, not yourself" discipline that culminates in the next phase.

Phase 33: Three Models Review Every Change

the cross-model eval reviewer

Goal: stop the AI that builds this product from being the only thing that checks the AI's work.

Almost everything in this product is built by an AI assistant, and for months the only review was that same assistant reading back its own diff. That is not a check, it is the same mind, in the same conversation, re-reading its own assumptions and nodding along. It misses whatever it was already blind to. And a different model is blind to different things: the systematic gaps in one are not the gaps in another.

What I Built

So I built a panel. Every pull request now gets reviewed, independently and in parallel, by three different AI models from three different companies. They never see each other's answers. When two or three flag the same thing, that agreement raises my confidence it is real; when only one catches something, that lone flag is the interesting signal, almost always one model's blind spot and increasingly often something the model that wrote the code simply did not see.

The part I care about most is not the review comment, it is the ledger behind it. Every review quietly appends a line to a running log: what it cost (real money, three paid APIs, pennies a time), what each model found and specifically what the OTHER models caught that the author model missed. Over months that log becomes a measured record of my AI's blind spots, instead of a vague sense that it "probably misses things."

Then the panel did the one thing that made me trust it: I pointed it at itself. Before the merge I had already had the author model review this code twice over, a multi-agent deep pass and a separate fresh-context reviewer, both the same model, both said ship. Then a different model looked once and found a real security hole all three earlier passes had walked right past: the piece that records the log trusted any comment carrying its marker, so anyone who could comment on a pull request could feed it a fake record and poison the very ledger I had built. It also caught that the tool was quietly shipping my whole code diff to two outside vendors with nothing redacted. Neither was hypothetical.

What I Learned

The uncomfortable part is why I missed them, because it is the same three reasons any author misses things in their own work. First, I trusted my own design: I wrote the code that posts that comment, so I read everything else assuming the comment was always mine and never once pictured someone faking one. Second, I framed my own review, so it could only check the risks I had already thought of; the other model arrived with no frame and asked the questions I had not, like "who else can write this?" and "where does this data actually go?". Third, every review pass was the same model, with the same instincts and the same blind spots. You cannot out-review your own blind spots by running more of the same reviewer.

The cheapest way to catch your own blind spots is to ask something that does not share them. Self-review, human or AI, is structurally weak because the reviewer and the author carry the same assumptions: diversity of reviewer beats effort of reviewer. And the highest-value output was not the pass-or-fail verdict, it was the log: turning "it probably misses things" into a growing, measured record of exactly what gets missed.

Drilling deeper on the "why"

The deepest point of this phase, and arguably the whole journey, is that effort and diversity are different axes, and I had been spending everything on the wrong one. Three review passes by the same model is three times the effort and zero additional diversity: it is the same mind reaching the same conclusions three times, each pass blind to exactly what the previous pass was blind to. A single pass by a genuinely different model added almost no effort and a whole new set of assumptions, and that is what found the two holes. A reviewer that fails loudly is merely annoying; a reviewer that fails silently is worse than none, because you trust it. The reason the panel earned my trust is that the first thing it reviewed was itself and it caught its own silent failures before they reached me.

Alternative considered The path I had actually been on was to keep increasing effort with the same reviewer: a deeper multi-agent pass, then a second fresh-context pass, both the same model. It feels like diligence, and it is genuinely more work. I replaced it with parallel review by three models from three companies because the failure I needed to catch was a blind spot, and a blind spot is, by definition, something more of the same eye cannot see. The cost is real, three paid APIs running on every pull request, pennies at a time, and I log that cost line by line so it stays honest. The benefit is the only thing that has ever actually found my author model's silent misses: a reviewer that does not share its assumptions.
Edge cases that bit The two holes the outside model found are textbook trust boundaries the author model could not see because it had drawn them itself. The first: the ledger trusted any comment bearing its marker, so it implicitly assumed the only writer was its own bot, when in fact anyone who can comment on a pull request can post a lookalike and poison the log, the exact record I built to measure trustworthiness. The second: the tool shipped the entire code diff to two third-party vendors with nothing redacted, a data-egress boundary I never consciously crossed because I never pictured the diff leaving. Both are "who else can act here, and where does this data go" questions, the questions an author with a frame does not ask about their own design.
What this unlocked This closed the loop the entire journey had been circling: for months the thing doing most of the building was the one thing I could not independently measure or check. The panel makes every future change reviewed by minds that do not share the author's blind spots, and the ledger turns "it probably misses things" into a measured, improving record of what actually gets missed. The proof it was worth building is that the very first thing it reviewed was itself, and it caught two silent bugs that would have quietly defeated its own purpose.
